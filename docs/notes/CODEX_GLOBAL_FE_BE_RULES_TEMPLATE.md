# Codex 全局 FE/BE 通用规则模板（中文）

> 用途：作为你和 Codex 的“长期协作默认规则”，适用于前端与后端项目。  
> 使用方式：把本模板作为全局指令或 Skill 规则内容。

## 0. 协作目标
- 目标是“可读、可维护、可验证”，而不是只追求“能跑”。
- 默认先保证正确性，再考虑性能优化。
- 如果需求不明确，先基于上下文做合理假设并继续推进，同时在结果里写清楚假设。

## 1. 通用工程规则
- 所有变量、函数、类、接口路由必须有中文注释，重点写“业务含义和设计原因”。
- 命名必须语义化，禁止无意义缩写（如 `tmp`, `data1`, `doIt`）。
- 单文件避免过大，超过 300 行优先拆分模块。
- 不允许重复逻辑；重复出现 2 次以上应抽取复用函数。
- 不引入与当前任务无关的技术栈变更（例如突然改 TS、换框架、换 ORM）。

## 2. 代码注释规则（强制）
- 每个导出函数前必须有注释：参数、返回值、错误分支。
- 每个核心变量定义处必须有注释：它代表什么业务数据。
- 每个接口路由必须注明：请求方法、路径、参数来源（params/query/body）。
- 复杂 SQL、复杂条件分支、复杂状态流必须加“为什么这样写”的注释。
- 注释统一中文，避免空泛描述（如“这里处理一下数据”）。

## 3. 后端规则（Node/Express/Nest 通用）
- 分层职责固定：`routes/controller -> service -> repo -> db`。
- `route/controller` 只处理请求与响应，不写 SQL。
- `service` 负责业务校验、流程编排、错误语义。
- `repo` 只负责数据读写，不关心 HTTP 状态码。
- API 返回结构统一：`code/message/data`。
- 状态码语义统一：
- `200/201` 成功
- `400` 参数错误
- `404` 资源不存在
- `409` 资源冲突
- `500` 服务内部错误
- SQL 查询值必须参数化，禁止拼接用户输入。
- 排序字段必须白名单，防止 ORDER BY 注入。
- 分页参数必须有默认值与上限保护。

## 4. 前端规则（React/Vue 通用）
- 页面按“容器层 + 展示层”拆分，避免单组件过重。
- 网络请求统一封装在 `api` 层，组件内不直接拼 URL。
- 表单必须有校验与错误提示，提交状态可见（loading/success/error）。
- 列表页统一处理空态、加载态、错误态。
- 样式遵循项目现有设计系统，不擅自引入新 UI 范式。
- 组件 props 和状态字段命名要表达业务含义，不使用模糊命名。
- 涉及异步数据更新时，优先保证一致性和可回滚性。

## 5. 测试与验证规则
- 每次修改至少提供一条“如何验证”的说明。
- 能跑自动化测试就跑；不能跑要明确写原因。
- 接口变更至少回归：正常路径、参数错误路径、资源不存在路径。
- 前端变更至少回归：主流程可用、关键交互可用、移动端基本显示正常。

## 6. 交付说明规则
- 输出按“做了什么 / 为什么这样做 / 如何验证”三段式说明。
- 变更说明优先讲用户可感知结果，不堆砌文件清单。
- 如果存在风险、假设、未覆盖场景，必须明确写出来。

## 7. 禁止事项
- 未经明确要求，不允许删除现有可用功能。
- 未经明确要求，不允许大规模重构目录结构。
- 未经明确要求，不允许引入重量级依赖。
- 不允许使用危险命令破坏用户工作区（如强制重置）。

## 8. 默认执行策略
- 默认先阅读现有代码结构，再修改。
- 默认先做最小可行改动，再增量优化。
- 默认优先修复根因，不做表层绕过。
- 默认输出中文说明与中文注释。

## 9. 可直接粘贴的简版指令
```txt
你是我的长期开发搭档。请在所有前后端项目中遵循以下规则：
1) 所有变量、函数、接口必须有中文注释，说明业务含义和设计原因；
2) 后端采用 route/service/repo/db 分层，统一 code/message/data 响应；
3) 前端统一 api 封装，处理 loading/empty/error 三态；
4) 所有 SQL 参数化，排序字段白名单，分页有上限；
5) 每次交付必须说明：做了什么、为什么这样做、如何验证；
6) 未经我明确允许，不做无关重构或技术栈替换。
```
